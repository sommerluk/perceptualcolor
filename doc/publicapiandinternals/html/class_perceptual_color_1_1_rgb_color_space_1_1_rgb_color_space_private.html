void PerceptualColor::RgbColorSpace::RgbColorSpacePrivate::deleteTransform ( cmsHTRANSFORM &  transformHandle)
static

Conveniance function for deleting LittleCMS transforms.

cmsDeleteTransform() is not comfortable. Calling it on a nullptr crashes. If called on a valid handle, it does not reset the handle to nullptr. Calling it again on the now invalid handle crashes. This conveniance function can be used instead of cmsDeleteTransform(): It provides some more comfort, by adding support for nullptr.

Parameters
transformHandlehandle to the transform
Postcondition
If the handle is nullptr, nothing happens. Otherwise, cmsDeleteTransform() is called and then, the handle is set to nullptr.

QPoint PerceptualColor::RgbColorSpace::RgbColorSpacePrivate::nearestNeighborSearch ( const QPoint  originalPoint,
const QImage &  image 
)
static

Search the nearest non-transparent neighbor pixel.

This implements a Nearest-neighbor-search.

Note
This code is an inefficient implementation. See here for a better approach.
Parameters
originalPointThe point for which you search the nearest neighbor, expressed in the coordinate system of the image. This point may be within or outside the image.
imageThe image in which the nearest neighbor is searched. Must contain at least one pixel with an alpha value that is fully opaque.
Returns
  • If originalPoint itself is within the image and a non-transparent pixel, it returns originalPoint.
  • Else, if there is are non-transparent pixels in the image, the nearest non-transparent pixel is returned. (If there are various nearest neighbors at the same distance, it is undefined which one is returned.)
  • Else there are no non-transparent pixels, and simply the point 0, 0 is returned, but this is a very slow case.

ChromaLightnessImage* PerceptualColor::RgbColorSpace::RgbColorSpacePrivate::m_nearestNeighborSearchImage

TODO WARNING This is a memory leak. This member is not deleted in the desctuctor of the class. This is because ChromaLightnessImage has a shared pointer to this object. If nearestNeighborSearchImage would be on the heap, deleting in when this object is deleted will call again deleting this object, and this recursive destructor call will crash the library. Therefore, we implement nearestNeighborSearchImage as a pointer with memory leak to avoid the crash. This memory leak is not fixed currently, because anyway we need to re-write all the code for nearest-neigbor search.
